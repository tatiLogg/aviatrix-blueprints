SCREEN 3: DCF Policies (Zero Trust Rules)

Navigate to: Security > Distributed Cloud Firewall > Rules

Show the 5 policies:

Priority 100 - allow-prod-to-db
  Source: Prod
  Destination: Database
  Action: PERMIT
  Purpose: Legitimate business need

Priority 110 - allow-dev-to-prod-read-only
  Source: Dev
  Destination: Prod
  Action: PERMIT (ICMP only)
  Purpose: Monitoring/health checks

Priority 200 - deny-dev-to-db
  Source: Dev
  Destination: Database
  Action: DENY
  Purpose: Blocks lateral movement

Priority 210 - deny-prod-to-dev
  Source: Prod
  Destination: Dev
  Action: DENY
  Purpose: Prevents reverse contamination

Priority 1000 - default-deny-all
  Source: Any
  Destination: Any
  Action: DENY
  Purpose: Zero Trust default

Key Message: "Notice policy 200—this explicitly blocks development from accessing the database. Even if an attacker compromises a dev server, they hit a wall. This is Zero Trust: never trust, always verify, least privilege by default."

Key Message: "FlowIQ shows us WHAT traffic is flowing. DCF Monitor shows us what the firewall DID with that traffic—permit or deny based on our Zero Trust policies. Together, this is complete visibility and control."

Anticipated Questions & Answers

QUESTION: "How does this compare to AWS Security Groups?"

ANSWER: "Security Groups are stateful L4 firewalls tied to individual instances. DCF operates at the transit layer, seeing all cross-VPC traffic. Plus, SmartGroups are dynamic—resources auto-join based on tags. With Security Groups, you'd need to manually update rules every time an IP changes."

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

QUESTION: "What happens if we add a new database server?"

ANSWER: "Just tag it with Environment=db. It automatically joins the db SmartGroup and inherits all policies. Production can reach it, development cannot—instantly, no configuration needed."

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

QUESTION: "Can we do this across AWS and Azure?"

ANSWER: "Yes! SmartGroups work across any cloud. You can have dev in AWS, prod in Azure, and the same policy 'deny dev to prod' applies. DCF abstracts away cloud-specific differences."

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

QUESTION: "How do we know it's working?"

ANSWER: "The Monitor tab shows real-time traffic with color-coded allow/deny. Plus, we can SSH to the test VMs and run connectivity tests. I can show you—prod can ping the database, but dev gets 'connection refused'."

QUESTION: "What about performance impact?"

ANSWER: "Minimal—sub-millisecond latency. Traffic flows through the transit gateway anyway for inter-VPC communication. DCF inspection adds negligible overhead since it's inline, not a proxy."
